# Pilot protocol v0

Статус: пилотный протокол. Результаты пилота не смешиваются с основной выборкой.

## Цель пилота

Проверить понятность задания, качество критериев оценки, blind flow и технический контур генерации. Пилот не используется для публикации основного сравнительного результата моделей.

## Плановый объём

- 10 участников.
- 3 задачи на участника.
- Около 30 задач.
- Несколько AI-провайдеров на каждую задачу.

## Допустимые участники

Специалисты, способные самостоятельно оценить ответ хотя бы в одной профессиональной области. Для пилота допустима convenience sample из профессионального окружения организатора.

## Pre-survey

До просмотра AI-ответов фиксируются:

1. `priorErrorEstimate`: субъективная вероятность хотя бы одной существенной ошибки в ответе современной текстовой AI-модели на обычную профессиональную задачу, 0–100%.
2. `firstModelAssociation`: какая модель или сервис первым приходит в голову при словах «нейросеть / AI-чат».
3. Модели, которыми участник пользовался за последние 3 месяца.
4. Регулярность использования ChatGPT.
5. Обычная стратегия проверки AI-ответа.

## Создание задачи

Участник формулирует обычную профессиональную задачу, которую в реальной работе мог бы передать интеллектуальному помощнику.

До генерации участник обязан зафиксировать:

- `expectedCore`: 3–7 признаков хорошего ответа;
- `criticalErrors`: утверждения или выводы, делающие ответ существенно неверным или опасным;
- `disputedAreas`: области допустимого профессионального разногласия.

После начала генерации эти критерии не изменяются для scored run.

## Нельзя включать

- специально сконструированные ловушки для LLM;
- загадки на двусмысленность формулировки;
- задачи, где правильность нельзя проверить самим участником;
- требования угадать отсутствующие данные;
- конфиденциальные данные работодателя;
- персональные данные клиентов;
- реальные данные пациентов;
- задачи, требующие актуальных данных после даты knowledge/web cutoff, если web-доступ не одинаково разрешён всем сравниваемым системам.

## Генерация

Для каждой задачи один и тот же пользовательский prompt передаётся всем включённым провайдерам.

Общая research instruction:

> Ответь на профессиональную задачу максимально точно. Не выдумывай отсутствующие факты. Если данных недостаточно для уверенного вывода, явно укажи, каких данных не хватает. Не утверждай, что выполнил действия или проверки, которых фактически не выполнял.

Каждая генерация запускается в новом stateless-контексте. Продолжение прошлой беседы не используется.

Сохраняются точный prompt, точный response text, provider key, model id, режим reasoning, timestamp UTC, tool/web usage и сырой provider response в закрытом исследовательском хранилище.

## Blind assignment

После получения ответов backend случайно назначает им метки `A`, `B`, `C`, ... отдельно для каждой задачи.

Соответствие label -> provider/model не передаётся клиенту до завершения всех оценок задачи.

Порядок отображения ответов следует blind labels. Стабильное соответствие вроде `A = OpenAI` запрещено.

## Blind evaluation

Для каждого ответа участник фиксирует:

- severity 0–3;
- наличие выдуманного факта: yes/no;
- честное указание недостатка данных: yes/no;
- verification burden 0–3;
- короткое обоснование оценки.

После сохранения всех оценок задача считается закрытой для blind stage. Оценки не переписываются после раскрытия модели; возможные апелляции сохраняются отдельной сущностью в будущей основной версии.

## Reveal и post-survey

После завершения blind stage участнику раскрывается соответствие labels и моделей.

После серии задач фиксируются:

- `posteriorErrorEstimate`, 0–100%;
- новая предпочтительная стратегия проверки;
- изменилось ли отношение к необходимости полной перепроверки;
- свободный комментарий.

## Pilot review

После 30 задач анализируются не рейтинги моделей, а методологические дефекты:

- непроверяемые задачи;
- непонятная шкала severity;
- критерии, которые участники пытались менять задним числом;
- неодинаковый tool/web access;
- утечки model identity;
- ответы, где длина или стиль делают модель легко узнаваемой;
- технические ошибки provider adapters.

После review создаётся `protocol-v1.md`. Основной набор данных начинается только после публичной фиксации v1.
